## 第二部分：技术栈选型理由

### 1. uv 而不是 conda/pip

**为什么选 uv：**
- uv 是用 Rust 编写的 Python 包管理器，安装速度比 pip 快 10-100 倍。
- 原生支持 `pyproject.toml`，不需要额外的 `requirements.txt`。
- 内置虚拟环境管理，一个命令 `uv run` 就能搞定环境创建和脚本执行。
- 确定性的依赖解析（lockfile），确保团队成员的环境完全一致。

**为什么不选 conda：**
- conda 体积庞大（Miniconda 也要几百 MB），对这种纯 Python 项目来说过于笨重。
- conda 的依赖解析速度慢，经常出现"Solving environment"卡住几分钟的情况。
- conda 混用 conda 和 pip 安装包时容易出现环境污染。

**为什么不选 pip：**
- pip 没有内置的虚拟环境管理，需要额外使用 venv。
- pip 没有 lockfile 机制（pip-tools 是第三方方案），无法保证可复现性。
- pip 的依赖解析器虽然在新版本有所改进，但速度仍远不及 uv。

### 2. yfinance 而不是 Alpha Vantage/Quandl

**为什么选 yfinance：**
- 完全免费，无 API key 要求，无请求频率限制。
- 直接返回 pandas DataFrame，和后续的数据处理管线无缝衔接。
- 数据质量够用：OHLCV + 已调整价格（auto_adjust）。
- 社区活跃，是 Python 金融数据获取的事实标准。

**为什么不选 Alpha Vantage：**
- 免费版每分钟 5 次请求、每天 500 次，批量拉取多个标的时很快就会触发限制。
- 需要注册获取 API key，增加了使用门槛。
- 返回 JSON 格式，需要额外的解析步骤。

**为什么不选 Quandl（现在叫 Nasdaq Data Link）：**
- 很多数据集已经变成付费的了。
- 免费的股价数据更新不够及时。
- API 设计偏向机构用户，对个人项目来说太重了。

### 3. ta 库而不是 TA-Lib

**为什么选 ta：**
- 纯 Python 实现，`pip install ta` 即装即用，没有任何编译依赖。
- API 设计现代化，面向对象接口（如 `ta.trend.MACD(close)`），代码可读性好。
- 覆盖了常用的 100+ 种技术指标。

**为什么不选 TA-Lib：**
- TA-Lib 底层是 C 库，安装时需要先编译 C 代码，在 macOS 上经常遇到 `brew install ta-lib` 后还要配置路径的问题，Windows 上更是噩梦。
- 虽然性能更好，但在这个项目的数据量级（几千行）下，性能差异可以忽略不计。
- TA-Lib 的安装问题是教学项目的第一大劝退因素。

### 4. XGBoost + LightGBM 而不是 LSTM/深度学习（重点）

**为什么选树模型集成：**

这是最关键的技术决策。很多初学者的直觉是"深度学习更厉害"，但在表格型金融数据上，梯度提升树通常优于深度学习，原因如下：

1. **数据量不足**：这个项目每个标的只有约 1000-1300 个样本。LSTM 等深度学习模型有百万级参数，在这种小数据集上几乎必然过拟合。而 XGBoost/LightGBM 通过限制树的深度和数量，天然抗过拟合。

2. **特征是人工设计的表格数据**：我们的输入不是原始时间序列，而是经过特征工程加工后的 30 个数值特征。树模型天生擅长处理这种异质的表格数据——不同特征的量纲不同、分布不同，树模型通过分裂点选择自然处理。LSTM 的优势在于从原始序列中自动学习时序模式，但我们已经用技术指标手动提取了这些模式。

3. **可解释性**：树模型可以输出 `feature_importances_`，告诉你哪些特征对预测最有用。在金融领域，一个你无法解释的"黑箱"模型是不敢用的——你不知道它到底在依赖什么信号做决策。

4. **训练速度**：XGBoost + LightGBM 训练 50 次随机搜索只需要几十秒，而 LSTM 的超参搜索可能需要几个小时。这对于快速迭代至关重要。

5. **学术界的共识**：大量 Kaggle 竞赛和学术论文（包括 2022 年的 "Why do tree-based models still outperform deep learning on tabular data?" ）都表明，在中小规模表格数据上，GBDT 类模型是统治性的。

**什么时候该用深度学习：**
- 数据量极大（百万级 tick 数据）
- 输入是原始时间序列或非结构化数据（新闻文本、K 线图像）
- 需要捕捉非常长期的时序依赖关系

### 5. React/Next.js + FastAPI 而不是 Streamlit/Flask/Django

**为什么选 Next.js 14 + FastAPI 的前后端分离架构：**

项目从最初的 Streamlit 单体应用升级为前后端分离架构，原因如下：

- **专业级 UI 体验**：React 生态提供了 TradingView Lightweight Charts（专业K线图）、Recharts（数据可视化）、Framer Motion（动画）等金融级组件，远超 Streamlit 的内置图表能力。
- **完全的 UI 控制权**：Tailwind CSS 自定义主题（黑色背景 + 红橙色 #E8522A 强调色 + 京华老宋体字体），打造独特的视觉风格，这在 Streamlit 中很难实现。
- **TypeScript 类型安全**：前端代码的类型检查在编译期捕获错误，提高代码质量。
- **FastAPI 高性能**：基于 ASGI 的异步框架，自动生成 OpenAPI 文档，REST API 接口清晰（`/api/symbols`、`/api/predict/{symbol}` 等）。
- **关注点分离**：前端负责展示和交互，后端负责数据处理和模型推理，ML pipeline（src/ 目录）完全不需要改动。
- **启动时自动刷新**：FastAPI 后端在启动时自动拉取最新数据并重新训练模型，无需手动操作。
- **预测周期可选**：支持 5/10/20 个交易日的预测周期切换，Streamlit 版本仅支持固定的 10 天。
- **一键启动**：`start.sh` 脚本同时启动后端和前端，简化部署流程。

**为什么不继续用 Streamlit：**
- Streamlit 的"每次交互重新执行整个脚本"的渲染模型在复杂场景下性能不佳。
- Streamlit 的样式自定义能力有限，难以实现专业金融终端的视觉效果。
- Streamlit 将前后端耦合在一起，不利于独立扩展和部署。
- Streamlit 的图表交互功能（如专业K线图、雷达图）不如专用的前端图表库丰富。

**为什么不选 Flask/Django：**
- Flask/Django 是同步框架，不如 FastAPI 的异步性能好。
- FastAPI 的自动 API 文档（Swagger UI）和 Pydantic 数据校验大幅减少样板代码。
- Django 对于一个纯 API 后端来说过于笨重。

### 6. StandardScaler 而不是 MinMaxScaler

**为什么选 StandardScaler（Z-score 标准化）：**
- 将特征转换为均值=0、标准差=1 的分布，对异常值更鲁棒。
- 树模型其实不需要标准化（因为分裂点只看排序），但在 RandomizedSearchCV 中统一使用 StandardScaler 可以让不同模型的超参搜索更稳定。
- 金融数据中经常出现极端值（暴涨暴跌），StandardScaler 不会把正常值压缩到很窄的范围。

**为什么不选 MinMaxScaler：**
- MinMaxScaler 把数据缩放到 [0, 1]，一个极端异常值就会把其他所有值压到很小的区间。
- 金融数据的分布通常有厚尾（fat tail），不适合用 MinMaxScaler。
- 新数据可能超出训练集的范围，导致 MinMaxScaler 输出超出 [0, 1]。

### 7. TimeSeriesSplit 而不是普通 train_test_split（重点：前视偏差）

**为什么选 TimeSeriesSplit：**

这是金融机器学习中最容易犯的致命错误——如果你用普通的 `train_test_split` 或 `KFold`，你会**用未来的数据训练模型，然后用过去的数据测试它**。

普通 KFold 的问题：
```
假设数据是 2021-2025 年的股价
KFold 可能把 2023 年的数据放进训练集，2022 年的放进验证集
→ 模型用 2023 年的信息去"预测" 2022 年 → 当然准！但这是作弊。
```

TimeSeriesSplit 的做法：
```
第 1 折: 训练 [2021-01 ~ 2022-06] → 验证 [2022-07 ~ 2023-01]
第 2 折: 训练 [2021-01 ~ 2023-01] → 验证 [2023-02 ~ 2023-08]
第 3 折: 训练 [2021-01 ~ 2023-08] → 验证 [2023-09 ~ 2024-03]
第 4 折: 训练 [2021-01 ~ 2024-03] → 验证 [2024-04 ~ 2024-10]
第 5 折: 训练 [2021-01 ~ 2024-10] → 验证 [2024-11 ~ 2025-06]
```

关键原则：**训练集永远在验证集之前，模型永远只用过去预测未来。** 这叫做"walk-forward validation"，是金融机器学习的行业标准。

项目还做了进一步的保护：在 `config.py` 中显式定义了 `TRAIN_END` 和 `VAL_END`，确保 train/val/test 三个集合严格按时间顺序排列。

## 第三部分：核心模块逐行解析

### 3.1 数据获取（data_fetcher.py）

#### yfinance 的工作原理

yfinance 并不是一个官方 API——它通过模拟浏览器请求，抓取 Yahoo Finance 网站的数据接口。核心调用 `yf.download(symbol, start, ...)` 会向 Yahoo 的内部 API 发送 HTTP 请求，返回 JSON 数据后解析为 pandas DataFrame。

`auto_adjust=True` 参数告诉 yfinance 返回**复权价格**（adjusted price），即将历史价格按股票分割（split）和分红（dividend）进行调整，使得价格序列在时间上可比。如果不做复权，一次 1:4 的拆股会让股价突然跌 75%，模型会把这当作暴跌信号。

#### 增量更新的逻辑

```python
if path.exists() and not force:
    df = pd.read_csv(path, ...)
    last_date = df.index[-1]
    new = yf.download(symbol, start=last_date + pd.Timedelta(days=1), ...)
    if not new.empty:
        df = pd.concat([df, new]).loc[~...index.duplicated(keep="last")]
        df.to_csv(path)
```

这段代码实现了"只拉新数据"的增量更新：
1. 如果 CSV 缓存已存在，读取它，找到最后一个日期。
2. 只从"最后日期+1天"开始请求新数据。
3. 用 `pd.concat` 拼接，并通过 `index.duplicated(keep="last")` 去重（保留最新版本）。
4. 把更新后的完整数据写回 CSV。

这样做的好处是避免每次都从 2021 年重新下载全部数据，节省时间和网络带宽。`force=True` 参数可以强制全量刷新。

#### CSV 缓存策略

项目用最简单的文件缓存：每个标的一个 CSV 文件（如 `data/QQQ.csv`）。选择 CSV 而不是数据库有几个原因：
- 数据量小（每个标的约 1300 行 x 6 列），CSV 完全够用。
- CSV 是人类可读的，可以用 Excel 打开检查。
- 不需要安装数据库软件，降低项目部署复杂度。
- pandas 原生支持 `read_csv` / `to_csv`，零额外依赖。

特殊字符处理：`_csv_path()` 函数中 `symbol.replace('^', '')` 是因为 VIX 的 ticker 是 `^VIX`，但 `^` 在文件名中可能导致问题，所以去掉。

#### 为什么要拉 VIX/TLT/UUP/SPY

这些是宏观因子（macro factors），反映了整体市场环境：

- **VIX（恐慌指数）**：衡量 S&P 500 期权的隐含波动率。VIX 高 = 市场恐慌，VIX 低 = 市场平静。股票通常在 VIX 飙升时下跌。
- **TLT（20年期国债 ETF）**：反映长期利率。债券价格和利率反向运动，TLT 上涨通常意味着资金从股市流向债市（risk-off）。
- **UUP（美元指数 ETF）**：强势美元通常压制美股（尤其是科技股），因为跨国公司的海外收入在汇率换算后缩水。
- **SPY（S&P 500 ETF）**：作为大盘基准，用来计算个股的相对强度。一只股票跑赢 SPY 说明它有 alpha。

单独看一只股票的技术指标是不够的——你需要知道整个市场在发生什么。同样一个"RSI=70"，在牛市和熊市中的含义完全不同。

### 3.2 特征工程（features.py）

#### 趋势类特征

**SMA（简单移动平均线）**

计算公式：SMA(n) = (P_1 + P_2 + ... + P_n) / n

项目使用了 5/10/20/50 日 SMA。不同周期捕捉不同的趋势：
- SMA_5：短线趋势（约一周）
- SMA_10：两周趋势
- SMA_20：月线趋势
- SMA_50：中期趋势（约一季度）

项目的巧妙之处在于不直接使用 SMA 值，而是使用 `Close/SMA` 的比率：
```python
df[f"Close_SMA_{w}_ratio"] = close / df[f"SMA_{w}"]
```
比率 > 1 表示股价在均线之上（偏多），< 1 表示在均线之下（偏空）。用比率而不是绝对值的好处是消除了股价量纲的影响——一个股价 500 美元的股票和一个 50 美元的股票，它们的 Close/SMA 比率是可比的。

**EMA（指数移动平均线）**

EMA 给近期数据更高的权重，公式：EMA_t = alpha * P_t + (1 - alpha) * EMA_{t-1}，其中 alpha = 2/(n+1)。

项目用了 12 日和 26 日 EMA，这正好是 MACD 的两个标准参数。

EMA 对价格变化的反应比 SMA 更灵敏。在趋势转折时，EMA 会更快发出信号。

**MACD（移动平均收敛散度）**

MACD 由三个部分组成：
- MACD 线 = EMA(12) - EMA(26)：快慢均线的差值
- 信号线 = MACD 线的 EMA(9)：MACD 线本身的平滑版
- 柱状图 = MACD 线 - 信号线：两者的差值

金融含义：MACD 线从下向上穿越信号线 = 买入信号（金叉），从上向下穿越 = 卖出信号（死叉）。柱状图从负转正表示动能转多。

#### 动量类特征

**RSI（相对强弱指数）**

计算步骤：
1. 计算每日涨跌幅
2. 分别计算过去 14 天的平均上涨幅度（AG）和平均下跌幅度（AL）
3. RS = AG / AL
4. RSI = 100 - 100/(1+RS)

RSI 的范围是 0-100：
- RSI > 70：超买区域，股价可能回调
- RSI < 30：超卖区域，股价可能反弹
- RSI ≈ 50：中性

直觉解释：如果过去 14 天几乎每天都在涨，RS 很大，RSI 接近 100；反之 RSI 接近 0。

**Stochastic Oscillator（随机震荡指标）**

%K = (Close - Low_14) / (High_14 - Low_14) * 100
%D = SMA(3) of %K

衡量当前收盘价在过去 14 天价格范围中的位置。%K = 80 意味着收盘价接近 14 天最高价，暗示可能超买。

**ROC（变化率）**

ROC(n) = (P_today - P_n_days_ago) / P_n_days_ago * 100

项目计算了 5 日和 10 日 ROC，直接反映短期价格动量。ROC > 0 表示价格在上涨。

**Williams %R**

和 Stochastic %K 类似，但公式取反：%R = (High_14 - Close) / (High_14 - Low_14) * (-100)

范围是 -100 到 0。-20 以上为超买，-80 以下为超卖。

#### 波动率类特征

**Bollinger Bands（布林带）**

- 中轨 = SMA(20)
- 上轨 = SMA(20) + 2 * STD(20)
- 下轨 = SMA(20) - 2 * STD(20)

数学原理：假设收益率服从正态分布，95% 的价格应该在均值 +/- 2 个标准差内。当价格突破上轨，说明涨幅异常（可能超买）；跌破下轨，说明跌幅异常（可能超卖）。

项目使用 `BB_pct` = (Close - Lower) / (Upper - Lower)，即当前价格在布林带中的相对位置（0-1 之间）。

**ATR（平均真实波幅）**

True Range = max(High - Low, |High - Close_prev|, |Low - Close_prev|)
ATR = SMA(14) of True Range

ATR 衡量每日的平均波动幅度，不区分方向。ATR 高的股票风险大。在仓位管理中，ATR 常用来设定止损距离。

**历史波动率（20日）**

```python
df["Volatility_20d"] = close.pct_change().rolling(20).std()
```

计算过去 20 个交易日收益率的标准差。和 ATR 不同，这里用的是收益率（百分比变化）而不是绝对价格变动，更适合跨股票比较。

#### 成交量类特征

**OBV（能量潮）**

规则很简单：
- 如果今天收盘 > 昨天收盘：OBV += 今日成交量
- 如果今天收盘 < 昨天收盘：OBV -= 今日成交量
- 如果收盘价不变：OBV 不变

OBV 的金融含义：如果股价上涨的同时 OBV 也上涨，说明上涨有成交量支撑（健康的涨势）。如果股价创新高但 OBV 没有，这叫"量价背离"，是趋势即将反转的信号。

注意：项目最终 drop 了 OBV 的绝对值（因为绝对值跨股票不可比），保留了其他成交量特征。

**Volume_ratio（成交量比率）**

```python
df["Volume_ratio"] = volume / volume.rolling(VOLUME_MA_WINDOW).mean()
```

当日成交量 / 过去 20 日平均成交量。比率 > 1 说明今天成交活跃（可能有大事发生），< 1 说明交投清淡。

#### 宏观因子

```python
df[f"{prefix}_Close"] = df[f"{prefix}_Close"].ffill()
df[f"{prefix}_ROC_5"] = df[f"{prefix}_Close"].pct_change(5)
```

对每个宏观标的，项目取其收盘价和 5 日 ROC：
- **VIX_Close / VIX_ROC_5**：恐慌程度及其变化趋势
- **TLT_Close / TLT_ROC_5**：债券市场动向
- **UUP_Close / UUP_ROC_5**：美元强弱
- **SPY_Close / SPY_ROC_5**：大盘走势

**QQQ_SPY_ratio**：纳斯达克 100 ETF 相对 S&P 500 的比率。这个比率上升 = 科技股跑赢大盘，下降 = 价值股或防御板块更强。对于 QQQ/GOOGL/PLTR 这些科技股来说，这是一个非常重要的情绪指标。

`ffill()`（前向填充）用于处理非交易日的缺失值——如果某个标的当天没有交易（比如 VIX 和股票的交易日历略有不同），就用前一天的值填充。

#### 时间特征

```python
df["DayOfWeek"] = df.index.dayofweek   # 0=周一, 4=周五
df["Month"] = df.index.month            # 1-12
```

- **星期几效应**：学术研究发现周一的平均收益率往往低于其他交易日（"Monday effect"），周五收盘前可能有获利了结。
- **月份效应**：1 月效应（January effect）、"Sell in May and go away" 等季节性模式。
- 虽然这些效应在现代市场中已经弱化，但作为特征让模型自己决定是否有用是合理的。

#### 目标变量

```python
forward_return = df["Close"].pct_change(FORWARD_DAYS).shift(-FORWARD_DAYS)
df["Target"] = (forward_return > 0).astype(int)
```

这两行代码值得仔细理解：

1. `pct_change(10)` 计算当前价格相对 10 天前价格的涨幅。
2. `shift(-10)` 将这个涨幅**向前移动 10 天**。效果是：对于 t 日这一行，Target 表示的是"从 t 日起算，未来 10 天股价是涨还是跌"。

**为什么选 10 天前瞻：**
- 太短（1-2 天）：噪音太大，随机波动占主导，模型很难学到有意义的模式。
- 太长（60+ 天）：预测太远的未来不现实，中间有太多不可预见的事件。
- 10 天（约 2 周）：平衡了信噪比和实用性，对应一个合理的持仓周期。

**为什么用二分类而不是回归：**
- 回归预测的是具体涨幅（如 +2.3%），精度要求极高，金融数据的噪音使得这几乎不可能。
- 二分类只需预测"涨还是跌"，容错空间更大。即使模型只有 53% 的准确率，配合合理的仓位管理，长期也可能盈利。
- 分类问题更适合使用 AUC、F1 等成熟的评估指标。

#### 前视偏差的防范

前视偏差（look-ahead bias）是金融回测中最危险的错误。项目通过以下方式防范：

1. **目标变量用 `shift(-FORWARD_DAYS)`**：明确标记这是"未来"数据。最后 10 行会因为 NaN 被丢弃（`dropna(subset=["Target"])`），确保不会用不完整的未来数据。
2. **TimeSeriesSplit 交叉验证**：训练数据永远在验证数据之前。
3. **Scaler 只 fit 训练集**：标准化的均值和标准差只来自训练数据，不泄露测试集的统计信息。
4. **严格的 TRAIN_END/VAL_END 日期划分**：手动定义时间边界，杜绝数据泄漏。
5. **特征全部基于历史数据计算**：SMA(20) 用的是过去 20 天的数据，不会"偷看"未来。

### 3.3 模型训练（model.py）

#### 三个模型的原理

**RandomForest（随机森林）—— Bagging 原理**

随机森林的核心思想是"三个臭皮匠，顶个诸葛亮"：

1. 从训练集中随机有放回地抽样（bootstrap），生成 N 个略有不同的训练子集。
2. 在每个子集上训练一棵决策树，每次分裂时只考虑随机选取的一部分特征（`max_features`）。
3. 最终预测取所有树的投票平均（分类任务用多数投票）。

为什么抗过拟合：
- 单棵决策树非常容易过拟合（可以把训练集拟合到 100% 正确）。
- 但每棵树的训练数据不同、看到的特征不同，所以各棵树的"错误模式"不一样。
- 多棵树投票后，随机的错误互相抵消，只有真正的信号被保留。
- 这就是 Bagging（Bootstrap Aggregating）的原理。

**XGBoost（极端梯度提升）—— Boosting 原理**

与 Bagging 不同，Boosting 是**串行**地训练弱学习器：

1. 第 1 棵树在原始数据上训练，产生预测误差（残差）。
2. 第 2 棵树专门学习第 1 棵树的残差——即"第 1 棵树哪里预测错了"。
3. 第 3 棵树学习前两棵树合计后的残差。
4. 以此类推，每棵新树都在修补前面所有树的错误。

XGBoost 的"梯度"指的是用损失函数的梯度（导数）来指导残差的拟合方向。它还加入了二阶导数信息和正则化项（`reg_alpha`、`reg_lambda`），使得优化更精确、更不容易过拟合。

**LightGBM —— 和 XGBoost 的区别**

LightGBM 也是梯度提升树，但在实现细节上有两个关键区别：

1. **Leaf-wise vs Level-wise 分裂策略**：
   - XGBoost 按层生长（level-wise）：先把树的同一层全部分裂完，再进入下一层。
   - LightGBM 按叶子生长（leaf-wise）：每次选择损失下降最大的那个叶子节点进行分裂。
   - Leaf-wise 更高效（同样的叶子数量下损失更低），但更容易过拟合，需要用 `num_leaves` 参数控制。

2. **直方图优化**：LightGBM 把连续特征值分桶（histogram），大幅减少分裂点的搜索次数，训练速度比 XGBoost 快 2-5 倍。

#### 超参数搜索空间的设计理由

```python
XGB_PARAMS = {
    "n_estimators": [100, 200, 300, 500],       # 树的数量
    "max_depth": [3, 5, 7, 9],                   # 树的最大深度
    "learning_rate": [0.01, 0.05, 0.1, 0.2],     # 学习率
    "subsample": [0.6, 0.7, 0.8, 0.9],           # 每棵树使用的样本比例
    "colsample_bytree": [0.5, 0.6, 0.7, 0.8],   # 每棵树使用的特征比例
    "reg_alpha": [0, 0.1, 1],                     # L1 正则化
    "reg_lambda": [1, 2, 5],                      # L2 正则化
}
```

设计逻辑：
- **n_estimators**：范围 100-500。太少学不够，太多训练慢且可能过拟合。配合 `learning_rate` 使用：学习率低需要更多的树。
- **max_depth**：范围 3-9。金融数据中特征之间的交互关系不太深，深度 3-5 通常就够。深度太大容易过拟合噪音。
- **learning_rate**：范围 0.01-0.2。越小越保守，每棵树的贡献越小，需要更多树来补偿，但泛化能力通常更好。
- **subsample + colsample_bytree**：引入随机性，减少过拟合。类似随机森林的思想。
- **reg_alpha + reg_lambda**：L1 和 L2 正则化。L1 倾向于把不重要的特征权重压到 0（特征选择），L2 倾向于让所有权重都小一些（防止某个特征权重过大）。

LightGBM 的 `num_leaves` 参数是独有的，取代 `max_depth` 作为控制模型复杂度的主要手段。`num_leaves = 31` 是默认值，大致对应 `max_depth ≈ 5`。

#### TimeSeriesSplit 5折交叉验证

```python
tscv = TimeSeriesSplit(n_splits=5)
```

5 折意味着数据被分成 6 段，每次验证用第 k+1 段，训练用前 k 段：

```
Fold 1: ████░░░░░░  Train[段1]  │ Val[段2]
Fold 2: ████████░░  Train[段1-2] │ Val[段3]
Fold 3: ████████████░░  Train[段1-3] │ Val[段4]
Fold 4: ████████████████░░  Train[段1-4] │ Val[段5]
Fold 5: ████████████████████░░  Train[段1-5] │ Val[段6]
```

关键特点：
- 训练集随着 fold 的增加而扩大（越来越多的历史数据）。
- 验证集始终在训练集之后，模拟"用历史预测未来"的真实场景。
- 最终的评估指标取 5 个 fold 的平均值，减少单次划分的偶然性。

#### StandardScaler 为什么只 fit 训练集

```python
scaler = StandardScaler()
scaler.fit(X_train)          # 只用训练集计算 mean 和 std
X_train_scaled = scaler.transform(X_train)
X_val_scaled = scaler.transform(X_val)    # 用训练集的参数变换
X_test_scaled = scaler.transform(X_test)  # 用训练集的参数变换
```

如果你在整个数据集上 fit scaler，均值和标准差会包含验证集和测试集的信息。这意味着训练集的标准化结果受到了"未来数据"的影响——这是一种微妙的数据泄漏。

虽然对树模型来说影响不大（树模型对单调变换不敏感），但这是一个好习惯。在实际部署时，你也不可能知道未来数据的统计分布。

#### 集成（Ensemble）的加权逻辑

```python
total_auc = sum(val_aucs.values())
weights = {name: auc / total_auc for name, auc in val_aucs.items()}

ensemble_proba = sum(
    w * model.predict_proba(X_test_scaled)[:, 1]
    for (name, model), w in zip(trained.items(), weights.values())
)
```

加权规则很直觉：**在验证集上 AUC 越高的模型，在集成中的话语权越大。**

假设三个模型的验证 AUC 分别是 0.55、0.58、0.57，那么权重大约是 0.324、0.341、0.335。表现最好的 XGBoost 得到稍多的权重。

为什么集成通常比单个模型好？因为不同模型的错误模式不同——RandomForest 擅长处理非线性关系，XGBoost 擅长捕捉渐进的模式，LightGBM 擅长高效地找到最佳分裂。三个模型的"盲区"不太一样，加权平均后互相弥补。

#### 模型序列化（joblib）

```python
joblib.dump(result, Path(MODEL_DIR) / f"{symbol}_models.pkl")
```

为什么用 joblib 而不是 Python 内置的 pickle：
- joblib 对大型 numpy 数组有优化的序列化（用内存映射），速度更快。
- scikit-learn 官方推荐使用 joblib 保存模型。
- `.pkl` 文件包含了模型权重、scaler 参数、特征名称等所有推理所需的信息。

### 3.4 回测引擎（backtest.py）

#### 仓位管理策略

```python
if prob > 0.6:
    target_pct = 1.0      # 满仓
elif prob > 0.5:
    target_pct = 0.5      # 半仓
else:
    target_pct = 0.0      # 空仓（全现金）
```

这是一个简单但合理的分级仓位策略：
- 模型很有信心（>60%）时全力押注。
- 模型稍有倾向（50-60%）时只投入一半资金。
- 模型不看好（<=50%）时完全退出。

每 `FORWARD_DAYS`（10 个交易日）检查一次是否需要调仓，和模型的预测周期一致。这样做避免了每天频繁交易产生的高额手续费。

#### 交易成本模拟

```python
cost = abs(delta_shares * price) * 0.001  # 0.1%
cash -= delta_shares * price + cost
```

每次交易扣除交易金额的 0.1% 作为成本，这包括了：
- 券商佣金（现在很多券商零佣金，但这里保守估计）
- 买卖价差（bid-ask spread）
- 滑点（slippage）：你想以 100 美元买入，实际可能以 100.05 成交

0.1% 的假设对美股大盘股来说是合理偏保守的（实际可能更低），但对小盘股或流动性差的品种可能偏乐观。

#### 评估指标的计算公式

**夏普比率（Sharpe Ratio）**

```python
sharpe = (strategy_returns.mean() * 252 - 0.05) / (strategy_returns.std() * np.sqrt(252))
```

公式：Sharpe = (R_p - R_f) / sigma_p

其中：
- R_p = 策略的年化收益率（日均收益 * 252 个交易日）
- R_f = 无风险利率（这里取 5%，对应当前美国国债收益率）
- sigma_p = 策略收益率的年化标准差（日标准差 * sqrt(252)）

夏普比率衡量的是"每承担一单位风险，能获得多少超额收益"。夏普 > 1 是不错的，> 2 是优秀的，> 3 极为罕见。

**最大回撤（Maximum Drawdown）**

```python
cummax = pf["value"].cummax()
drawdown = (pf["value"] - cummax) / cummax
max_drawdown = drawdown.min()
```

最大回撤 = 从历史最高点到最低点的最大跌幅百分比。

例如：组合从 $12,000 跌到 $9,600，最大回撤 = (9600 - 12000) / 12000 = -20%。

最大回撤告诉你"最坏的情况下你会亏多少"。对投资者的心理承受力来说，这比平均收益率更重要。

**年化收益率**

```python
annual_factor = 252 / max(n_days, 1)
annual_return = (1 + total_return) ** annual_factor - 1
```

将测试期间的总收益换算为年化收益。比如 3 个月赚了 5%，年化约为 (1.05)^4 - 1 ≈ 21.6%。

#### 为什么和买入持有做对比

买入持有（Buy & Hold）是最简单的投资策略——第一天满仓买入，持有到最后一天。它是衡量任何主动策略是否"值得"的基准线。

如果你的 ML 策略跑不赢简单的买入持有，那所有的模型训练、特征工程都是徒劳的——你还不如什么都不做。

项目初始化 `$10,000`，同时在买入持有策略中买入等值的股票，两条曲线放在一起对比，非常直观。

#### 前端核心组件解析

**CandlestickChart**：基于 TradingView Lightweight Charts 的K线图组件。与 Plotly 的 `go.Candlestick` 相比，TradingView 图表提供了更接近专业交易软件的体验——内置时间轴导航、十字准线、实时价格标记等功能，渲染大量数据时的性能也显著更好。

**PredictionCard**：展示模型的预测结果，包括预测方向（涨/跌）、集成概率、以及基于概率的仓位建议。支持 5/10/20 天预测周期切换。

**ModelSignals**：汇总三个子模型（RF、XGB、LGBM）各自的预测信号和权重，让用户看到集成决策的"内部分歧"。

**RadarChart**：将多个评估维度（AUC、准确率、F1、夏普比率等）展示在雷达图上，一目了然地看出模型在哪些方面强、哪些方面弱。

**BacktestChart**：对比展示策略曲线和买入持有曲线，使用 Recharts 的面积图实现，支持悬停查看具体数值。

**FeatureImportance**：水平条形图展示特征重要性排序，帮助用户理解模型在依赖哪些信号做决策。

#### 一键启动脚本（start.sh）

`start.sh` 脚本同时启动 FastAPI 后端（默认端口 8000）和 Next.js 前端（默认端口 3000）。用户只需执行 `./start.sh`，无需分别手动启动两个服务。脚本会处理进程管理，确保退出时两个服务都被正确关闭。

## 第四部分：50 个核心问题（带答案）

### 数据相关（1-10）

**1. 为什么选 5 年历史数据而不是更长或更短？**

项目从 2021-01-01 开始，到现在约 4-5 年数据。选择这个范围是基于几个权衡：太短（1-2 年）的数据样本量不够，模型容易过拟合；太长（10-20 年）的数据包含了完全不同的市场结构（如 2008 金融危机时期），那时的市场规律未必适用于今天。5 年覆盖了牛市（2021）、熊市（2022）和恢复期（2023-2025），让模型有机会学到不同市场环境下的模式。另外，PLTR 是 2020 年 9 月才上市的，选 2021 年开始可以保证三个标的都有完整数据。

**2. OHLCV 中每一列的金融含义？**

OHLCV 是金融数据的基本格式：Open（开盘价）是每天第一笔交易的价格，反映了隔夜消息被消化后的市场预期；High（最高价）和 Low（最低价）代表当天的价格波动范围，是波动率指标的基础；Close（收盘价）是最后一笔交易的价格，是最重要的价格——因为它反映了交易日结束时市场的"最终共识"；Volume（成交量）代表当天的交易股数，反映市场的活跃度和参与者的兴趣。所有技术指标都基于这五个基础数据计算。在分析中，收盘价通常被赋予最高权重。

**3. auto_adjust=True 在 yfinance 中做了什么？**

`auto_adjust=True` 让 yfinance 返回经过除权除息调整的价格。当一家公司发放股息或进行股票分割时，股价会出现不连续的跳变。例如 GOOGL 在 2022 年 7 月进行了 1:20 的拆股，股价从约 2200 美元降到约 110 美元。如果不调整，模型会看到一个 95% 的"暴跌"信号，但实际上投资者的持仓价值没有变化。复权价格将历史上所有的分割和分红都反算回去，使得整个价格序列在时间维度上是连续可比的。这是做量化分析的基本前提。

**4. 为什么需要宏观数据（VIX/TLT/UUP）？**

个股不是在真空中交易的，它的涨跌受到整体市场环境的强烈影响。VIX（恐慌指数）衡量市场的整体恐惧程度，VIX 飙升时几乎所有股票都会下跌。TLT（长期国债 ETF）反映利率环境，利率上升压制成长股估值（因为未来现金流的折现率增加）。UUP（美元指数 ETF）影响跨国公司的利润，强美元对出口型企业不利。SPY 代表整体市场方向。把这些宏观因子加入模型，相当于告诉模型"当前的大环境是什么样的"，帮助它做出更全面的判断。

**5. CSV 缓存的增量更新是怎么实现的？**

实现逻辑分为三步：首先检查 CSV 文件是否存在。如果存在，读取已有数据并找到最后一个日期；然后只向 Yahoo Finance 请求该日期之后的新数据；最后将新旧数据拼接，去重后写回 CSV。去重用的是 `index.duplicated(keep="last")`，意思是如果新旧数据有重叠日期，保留最新版本。这种设计的好处是避免每次运行都重新下载全部历史数据，大幅减少了网络请求和等待时间。如果需要从头刷新数据，传入 `force=True` 即可跳过缓存。

**6. 数据缺失（NaN）是怎么处理的？**

NaN 在这个项目中有三个来源：第一，技术指标的"预热期"——SMA(50) 需要前 50 天的数据才能计算，所以前 50 行的 SMA_50 是 NaN。第二，目标变量——最后 10 行没有完整的未来 10 天数据，所以 Target 是 NaN。第三，宏观数据的日期不对齐——某些交易日可能某个标的没有数据。项目对此的处理是：宏观数据用 `ffill()` 前向填充（用最近的有效值），目标变量的 NaN 行直接丢弃（`dropna(subset=["Target"])`），剩余的 NaN 行在最后通过 `mask = X.notna().all(axis=1)` 统一过滤掉。

**7. 为什么不用数据库而用 CSV？**

对于这个项目的数据规模，CSV 是最优选择。每个标的约 1300 行 x 6 列，总计不到 1MB，CSV 读写在毫秒级完成。引入数据库（如 PostgreSQL 或 SQLite）会增加安装配置的复杂度、代码量（需要写 ORM 或 SQL 查询），以及调试难度——而性能上没有任何收益。CSV 还有一个巨大优势：可以直接用 Excel 打开检查数据质量，debug 时非常方便。当数据量达到百万行以上或需要多进程并发读写时，才值得切换到数据库。

**8. yfinance 的数据延迟有多大？**

yfinance 提供的日线数据延迟大约 15-30 分钟。在美股收盘后（东部时间 16:00 后），当天的数据通常在 30 分钟内可以获取。盘中拿到的数据可能是前一天的。对于这个项目来说不是问题，因为模型的预测周期是 10 个交易日，几十分钟的延迟无关紧要。但如果要做日内交易（intraday），yfinance 的延迟就太大了，需要使用实时数据 API（如 Alpaca、Interactive Brokers）。

**9. 为什么选 QQQ 而不是直接用纳斯达克指数 ^IXIC？**

QQQ 是一个真实可交易的 ETF（纳斯达克 100 追踪基金），而 ^IXIC 是一个指数，你不能直接买卖指数。项目的回测模块需要模拟"买入卖出"，所以必须使用可交易的标的。另外，QQQ 有真实的成交量数据，可以用来计算成交量类特征（OBV、Volume_ratio），而指数的成交量数据意义不大。QQQ 追踪的是纳斯达克 100（最大的 100 家非金融公司），成分和 ^IXIC（3000+ 家公司）有差异，但走势高度相关，作为科技板块代表完全够用。

**10. 数据量（1300行）够训练模型吗？**

对于深度学习来说，1300 行远远不够（LSTM 通常需要数万行以上）。但对于树模型来说，这个数量是勉强够用的。每棵决策树只需要找到几个最佳分裂点，1300 个样本足以让分裂点的选择有统计意义。不过，数据量偏少确实会带来一些问题：交叉验证的每个 fold 只有两百多个样本，评估指标的方差较大；模型可能无法充分学到一些低频模式（如"VIX 突破 40 后的行情"，这种情况在 5 年数据中可能只出现过 1-2 次）。增加数据量是改进项目的第一个方向。

### 特征工程相关（11-20）

**11. 为什么不能直接把股价塞进模型？**

直接用股价有两个致命问题。第一，股价本身是非平稳的（non-stationary）——GOOGL 从 100 涨到 200 再涨到 300，模型在 100 美元时学到的"规律"到 300 美元时完全不适用。第二，不同股票的价格量级差异巨大——QQQ 约 500 美元，PLTR 约 25 美元，直接比较毫无意义。技术指标把价格转换成了"状态描述"：RSI=70 无论股价是 50 还是 500，含义都是"超买"。Close/SMA 比率、ROC 等指标消除了量纲，使得模型学到的模式可以跨时间、跨股票泛化。

**12. SMA 和 EMA 的区别是什么？**

SMA（简单移动平均）对窗口内所有数据点赋予相同权重。EMA（指数移动平均）给近期数据更高的权重，权重按指数衰减。直觉上，EMA 对最新价格变化更敏感——如果股价突然暴跌，EMA 会比 SMA 更快地"反应过来"。这意味着 EMA 产生的趋势信号更及时，但也更容易被短期噪音误导（假信号更多）。项目同时使用两者，让模型自己决定在什么情况下应该更关注哪一个。SMA 更稳定适合判断中长期趋势，EMA 更灵敏适合捕捉短期转折。

**13. RSI 的计算公式和直觉解释？**

RSI 的计算分四步：(1) 计算每日涨跌幅；(2) 将上涨日的涨幅取平均（AG），下跌日的跌幅取平均（AL）；(3) RS = AG / AL；(4) RSI = 100 - 100/(1+RS)。直觉解释：如果一只股票连续 14 天都在涨（AG 很大，AL 接近 0），RS 趋向无穷大，RSI 趋向 100。反之 RSI 趋向 0。RSI=70 意味着最近上涨的力度是下跌力度的 2.33 倍（70 = 100 - 100/(1+2.33)），市场可能过度乐观。RSI 的核心思想是：任何趋势都不可能无限延续，当市场过度偏向一方时，均值回归的力量会发挥作用。

**14. MACD 三条线分别代表什么？**

MACD 线 = EMA(12) - EMA(26)：反映短期趋势和中期趋势的差异。当 MACD > 0，短期均线在中期均线之上，说明近期涨势强于中期。信号线 = MACD 线的 9 日 EMA：是 MACD 线本身的平滑版，用来过滤 MACD 的短期波动。柱状图 = MACD - 信号线：两者的差值，直观显示动能强度。柱状图从负转正（零轴以下向上穿越）= MACD 金叉，是经典的买入信号。反之为死叉卖出信号。MACD 本质上是一个"趋势的趋势"指标——它告诉你趋势是在加速还是减速。

**15. 布林带的数学原理是什么？**

布林带基于统计学的正态分布假设。中轨是 20 日均线（代表"正常"价格），上下轨各偏移 2 个标准差。在正态分布下，95.4% 的数据落在均值 +/- 2 倍标准差内。所以布林带表达的是：如果股价正常波动，它应该 95% 的时间在这个带内。当股价突破上轨，说明涨幅超过了"正常波动"范围，可能是超买；跌破下轨同理。布林带还有一个重要特征：带宽会自动收缩和扩张。当市场平静时（标准差小），带子变窄，暗示一个大波动可能即将到来（"布林带挤压"）。

**16. 什么是前视偏差？这个项目怎么避免的？**

前视偏差（look-ahead bias）是指在构建模型或回测时，不小心使用了在当时不可能获得的"未来信息"。最常见的例子：用包含未来数据的统计量做特征标准化、在交叉验证中把未来数据混入训练集。这个项目通过五层防线避免前视偏差：(1) 目标变量用 `shift(-10)` 明确标记为未来数据，最后 10 行因 NaN 被丢弃；(2) TimeSeriesSplit 保证训练集在时间上先于验证集；(3) StandardScaler 只在训练集上 fit；(4) 显式的 TRAIN_END/VAL_END 日期边界；(5) 所有技术指标只使用历史窗口内的数据（如 SMA(20) 用过去 20 天）。

**17. 为什么要做特征标准化？**

不同特征的数值范围差异巨大：RSI 在 0-100，ATR 可能在 0.5-5，Volume_ratio 在 0.2-3。虽然树模型理论上不需要标准化（因为决策树只关心特征值的排序），但在这个项目中标准化仍然有意义：(1) RandomizedSearchCV 中使用的评估指标（AUC）对概率校准有要求，标准化后的输入有助于模型产生更好的概率估计；(2) 统一处理方便代码管理；(3) 养成好习惯——如果后续加入逻辑回归或 SVM 等需要标准化的模型，不需要改动 pipeline。

**18. 30 个特征会不会太多导致过拟合？**

30 个特征对于 1300 个样本来说确实偏多（样本/特征比约 43:1），但项目通过以下机制控制过拟合：(1) 树模型的 `max_features` 参数让每棵树只看一部分特征（如 30%），相当于自动降维；(2) L1 正则化（`reg_alpha`）可以将不重要特征的权重压到零；(3) 交叉验证根据验证集性能选择超参数，过拟合的模型在验证集上表现会差；(4) 集成三个不同的模型进一步分散风险。不过，如果要进一步改进，可以考虑用特征重要性排序做特征选择，或者用 PCA 降维。

**19. 为什么加入"星期几"和"月份"作为特征？**

金融市场存在日历效应（calendar effect），这是行为金融学的经典发现。周一效应（Monday effect）：周末积累的利空消息在周一集中释放，导致周一平均收益率偏低。周五效应：交易者在周末前倾向于平仓锁定利润。一月效应：机构投资者在年底为了"窗口美化"（window dressing）卖出亏损股，一月再买回来，推高小盘股。"Sell in May"：5-10月的平均收益低于11-4月。虽然这些效应随着市场效率提高在逐渐减弱，但把它们作为特征让模型自己判断是否有用，成本很低。

**20. OBV（能量潮）为什么能反映市场情绪？**

OBV 的核心假设是"量在价先"——聪明的机构投资者会在价格大幅变动之前悄悄建仓或减仓，这会反映在成交量上。如果股价平稳但 OBV 持续上升，说明有大量买盘在"吸货"，价格可能即将上涨。反之，股价维持高位但 OBV 下降（"量价背离"），说明有人在高位出货。OBV 把成交量从一个绝对数字转化为一个累积趋势指标，让成交量也能像价格一样画出"趋势线"。不过项目最终丢弃了 OBV 的绝对值（因为跨股票不可比），主要保留了 Volume_ratio 这个标准化的成交量指标。

### 模型相关（21-35）

**21. 随机森林和单棵决策树有什么区别？**

单棵决策树就像一个"独裁者"——它做所有决策，任何训练数据的微小变化都可能导致完全不同的树结构，方差极大。随机森林就像一个"委员会"——数百棵树独立投票，最终取多数意见。每棵树都在略有不同的数据子集和特征子集上训练，所以它们各自有不同的"偏见"和"盲区"。投票平均后，随机的错误被抵消，只有在多数树上一致出现的模式（即真正的信号）被保留。这就是"群体智慧"（wisdom of crowds）的数学体现。方差减小了，偏差基本不变，整体预测更稳定。

**22. Bagging 和 Boosting 的核心区别？**

Bagging（如 RandomForest）是**并行**训练多个弱学习器，然后投票平均。每个学习器是独立的，互不影响。核心目标是**减少方差**（variance reduction）。Boosting（如 XGBoost、LightGBM）是**串行**训练，每个新学习器专注于修正前面学习器的错误。核心目标是**减少偏差**（bias reduction）。打个比方：Bagging 像让 100 个人独立做同一道题，取平均答案；Boosting 像一个人做题、改错、再做题、再改错，循环迭代。Boosting 通常能达到更低的训练误差，但也更容易过拟合，所以需要正则化参数来控制。

**23. XGBoost 的"梯度提升"是什么意思？**

"梯度"指的是损失函数对当前预测值的导数（梯度），它告诉我们"当前的预测差在哪里，差多少"。传统的提升树直接拟合残差（实际值 - 预测值），XGBoost 更进一步，用损失函数的一阶和二阶导数来指导下一棵树的生长方向。这相当于做了一次牛顿法优化，比简单的梯度下降更精确。形象地说：传统 Boosting 只知道"我预测错了多少"，XGBoost 还知道"错误的曲率"，能更精准地修正。加上正则化项（控制树的复杂度），XGBoost 在保持强大拟合能力的同时抑制了过拟合。

**24. LightGBM 比 XGBoost 快在哪里？**

LightGBM 的速度优势来自三个技术创新：(1) **直方图优化**：将连续特征离散化为 256 个桶，分裂点搜索从 O(#data) 降到 O(#bins)，大幅减少计算量。(2) **Leaf-wise 分裂**：不像 XGBoost 按层分裂（一层的所有节点都分裂），而是每次只分裂增益最大的一个叶子节点，同样的叶子数下需要更少的分裂操作。(3) **GOSS（Gradient-based One-Side Sampling）**：保留所有梯度大的样本（预测错得离谱的），随机采样梯度小的样本（已经预测得很好的），减少训练数据量但保持信息量。对于这个项目 1300 行的数据量，速度差异不明显，但如果数据量达到百万级，LightGBM 可以快 5-10 倍。

**25. 为什么不用深度学习（LSTM/Transformer）？**

五个理由：(1) 数据量不足——1300 行数据训练一个几万参数的 LSTM 几乎必定过拟合，而树模型只有几百个参数（分裂点）。(2) 输入已经是人工提取的表格特征，不是原始时间序列——LSTM 的优势在于自动学习序列模式，但我们已经用技术指标做了这件事。(3) 深度学习是黑箱，无法解释"为什么做这个预测"。(4) 训练时间长，超参搜索成本高。(5) 大量学术和工业实践表明，在小规模表格数据上，GBDT 类模型的性能等于甚至优于深度学习。只有在数据量极大、输入为原始序列或非结构化数据时，深度学习才有优势。

**26. 过拟合在金融预测中为什么特别危险？**

金融数据有两个特殊性质使过拟合格外致命：第一，信噪比极低——股价的短期波动大部分是随机噪音，真正的可预测信号非常微弱。一个过拟合的模型实际上在"记忆噪音"，而不是学习信号。第二，市场环境在不断变化（非平稳性）——2022 年的加息周期和 2023 年的 AI 牛市完全不同，过拟合到某一段历史的模型在新环境中可能表现灾难性。更危险的是，过拟合的模型在回测中表现极好（因为回测数据就是它"记忆"的数据），给你虚假的信心，导致真金白银的亏损。

**27. 正则化参数（reg_alpha, reg_lambda）在做什么？**

`reg_alpha` 是 L1 正则化系数，给损失函数加上叶子节点权重绝对值之和的惩罚。效果是将不重要特征的权重压到零，起到特征选择的作用。`reg_lambda` 是 L2 正则化系数，给损失函数加上叶子节点权重平方和的惩罚。效果是让所有权重都保持较小，防止模型过度依赖某个特征。直觉上：没有正则化的模型会为了拟合训练数据中的噪音而给某些特征很大的权重，正则化相当于说"除非这个特征真的很重要，否则不要给它太大权重"。项目搜索空间中 `reg_alpha` 在 [0, 0.1, 1]、`reg_lambda` 在 [1, 2, 5] 之间，由交叉验证自动选择最佳组合。

**28. TimeSeriesSplit 和普通 KFold 的关键区别？**

普通 KFold 随机打乱数据分成 K 份，任何一份都可能作为验证集。问题是：2024 年的数据可能进入训练集，2023 年的数据作为验证集——模型"用未来预测过去"，性能虚高。TimeSeriesSplit 保证时间上的单调性：训练集永远在验证集之前，而且每一折的训练集都是前面所有折的超集（越来越多的历史数据）。这模拟了真实的投资场景——你只有到"今天"为止的数据，用它预测"明天"。如果一个模型在 TimeSeriesSplit 下表现好，至少说明它在样本外有一定的泛化能力。

**29. RandomizedSearchCV 和 GridSearchCV 的优劣？**

GridSearchCV 穷举所有参数组合。以 XGBoost 为例，7 个参数的网格搜索有 4*4*4*4*4*3*3 = 9216 种组合，每种还要跑 5 折 CV，总计 46080 次模型训练——即使每次只需 1 秒，也要 12 小时。RandomizedSearchCV 只随机采样 N 组参数（项目设 `n_iter=30`），30*5 = 150 次训练，几分钟搞定。研究表明，在大多数情况下，随机搜索 60 次就能找到 95% 以上的最优解——因为通常只有 2-3 个参数真正关键，其余参数的影响很小。随机搜索还有一个优势：它可以搜索连续分布（不限于离散网格点），虽然这个项目没有用到这个特性。

**30. 为什么集成多个模型通常比单个模型好？**

集成学习的理论基础是"多样性"（diversity）。如果多个模型的错误是不相关的（一个模型在 A 类情况下预测错误，另一个在 B 类情况下预测错误），那么它们的平均预测就能互相补偿。RandomForest 擅长处理非线性关系，XGBoost 擅长捕捉渐进的模式，LightGBM 的 leaf-wise 策略可能发现不同的分裂路径。三个模型在不同的"角度"看同一个问题。加权平均后，共性信号被放大，个体噪音被抑制。实际效果是：集成模型的性能下限更高（最差也不会比最好的单个模型差太多），在大多数情况下还能小幅超越最佳单模型。

**31. 模型的准确率只有 50-55%，这真的有用吗？**

在金融预测中，这已经是有意义的了。想象一个简单的策略：只在模型预测"涨"时买入，否则持有现金。即使准确率只有 53%，意味着 100 次预测中对 53 次、错 47 次。如果每次对的时候赚 1%、错的时候也亏 1%，净收益是 (53-47)*1% = 6%。再考虑到"涨的时候涨得多、跌的时候跌得少"的不对称性（模型可能在大涨前更倾向于预测涨），实际收益可能更高。关键不在于准确率本身，而在于准确率和赔率的组合（即凯利公式的思想）。当然，53% 并不能保证赚钱——交易成本、滑点、连续亏损期的心理压力都是挑战。

**32. AUC 0.55 在金融领域算什么水平？**

AUC = 0.5 意味着模型和随机猜测一样好（没有任何区分能力）。在医学诊断或欺诈检测中，AUC 0.55 几乎没有用。但在金融预测中，AUC 0.55 已经是"有信号"的水平。原因是金融市场接近有效市场假说——大量聪明人在同时试图预测，可利用的信号被快速消化，剩余的可预测性非常微弱。学术论文中，能在日线级别的股票方向预测上达到 AUC 0.55-0.58 就是值得发表的结果。实际的量化对冲基金并不追求高 AUC，而是把微弱的信号通过高频交易或杠杆放大。对这个项目来说，AUC > 0.53 就值得进一步探索。

**33. Precision 和 Recall 哪个对交易更重要？**

取决于策略类型。如果你是"稳健型"——只在非常有把握时才交易，宁可错过机会也不想亏钱——那 Precision 更重要。高 Precision 意味着"模型说涨的时候大概率真的涨"，假阳性少。如果你是"进攻型"——不想错过任何一次上涨机会——那 Recall 更重要。高 Recall 意味着"几乎每次真正的涨都被模型捕捉到了"，但可能有更多假信号。对于这个项目的仓位管理策略（只有概率 > 0.5 才建仓），Precision 更关键——因为误判"涨"会导致亏损，而误判"跌"只是少赚（持现金不亏钱）。F1 分数是两者的调和平均，是折中指标。

**34. 为什么 scaler 只在训练集上 fit？**

这是一个经常被忽视但极其重要的原则。如果你在整个数据集上 fit scaler，均值和标准差会包含验证集和测试集的信息。举个极端例子：假设训练集的 RSI 均值是 50，测试集有一段 RSI 持续在 80+ 的超买期。如果在全集上 fit，均值会被拉到 55，训练集的标准化结果被"来自未来的信息"扭曲了。虽然对树模型影响较小（因为树只看排序），但这是一种数据泄漏，对其他类型的模型（如逻辑回归、SVM）影响显著。更重要的是，在实际部署时，你根本不可能知道未来数据的分布，所以 scaler 只能在已有数据上 fit。

**35. 模型能预测黑天鹅事件吗？**

不能，这是这类模型最根本的局限。黑天鹅事件（如新冠疫情、俄乌战争、硅谷银行倒闭）的定义就是"历史上从未发生过或极少发生的极端事件"。机器学习模型只能从历史数据中学习模式——如果一种模式从未出现在训练数据中，模型就无法识别它。更糟糕的是，黑天鹅事件发生时，所有正常的统计关系都会崩溃（如 VIX 和股价的关系在极端恐慌时会变得非线性），模型之前学到的"规律"在这种环境下可能完全失效。这就是为什么任何量化策略都必须有风控措施（如止损、仓位限制），模型只是决策的参考之一。

### 回测相关（36-42）

**36. 回测和实盘的核心区别？**

回测是在历史数据上模拟交易，实盘是用真金白银交易。两者有几个关键区别：(1) 滑点——回测假设你能以收盘价成交，但实盘中你可能以更差的价格成交（尤其是大单或流动性差的时候）。(2) 市场影响——回测中你的交易不会影响价格，但实盘中大笔买入会推高价格（尤其是小盘股）。(3) 延迟——回测中信号产生和执行是同时的，实盘中有计算时间、网络延迟。(4) 心理压力——连续亏损三次后，你还能坚持按模型操作吗？回测没有这个问题。(5) 制度变化——回测假设未来和过去一样，但市场结构在不断变化。

**37. 什么是过拟合回测？**

过拟合回测（backtest overfitting）是指通过反复调整策略参数来"优化"回测结果，直到得到漂亮的回测曲线。这和模型的过拟合本质相同：你不是在学习市场规律，而是在记忆特定历史数据的噪音。常见的做法：试了 100 种参数组合，选了回测表现最好的那组——但这组参数很可能只是偶然在这段历史数据上表现好。项目通过以下方式缓解：(1) 用交叉验证选参数，而不是在测试集上选；(2) 测试集完全留出，只在最后评估一次；(3) 回测使用简单且固定的仓位管理规则，没有过多可调参数。

**38. 夏普比率为什么是风险调整收益的标准指标？**

夏普比率回答了一个核心问题："承担额外风险是否值得？"如果你可以把钱存银行赚 5%（无风险利率），那你投资股票承担波动风险就应该赚更多。夏普比率衡量的是"每承担一单位波动风险（标准差），你比无风险利率多赚了多少"。比较两个策略时，仅看收益率是不够的——年化 30% 但最大回撤 50% 的策略，不一定比年化 15% 但最大回撤 10% 的策略好。夏普比率把风险因素纳入了比较。一般来说，夏普 > 1 是可接受的，> 2 是优秀的。项目中用无风险利率 5%（对应当前美国短期国债收益率水平）。

**39. 最大回撤为什么比总收益更重要？**

因为回撤直接影响投资者的生存能力和心理承受力。一个总收益 100% 但中途最大回撤 60% 的策略，意味着你的账户曾经从高点跌去了 60%——10 万变成 4 万。即使最终回到 20 万，99% 的个人投资者在账户缩水 60% 时就已经恐慌性平仓了。更关键的数学问题是：亏 50% 后需要赚 100% 才能回本（不是 50%）。最大回撤还暗示了策略的尾部风险——如果历史最大回撤是 30%，未来可能出现更大的回撤。专业的量化基金经理通常把最大回撤控制在 15-20% 以内，否则投资者会赎回基金。

**40. 0.1% 交易成本假设合理吗？**

对于美股大盘 ETF（如 QQQ、SPY），0.1% 是一个合理偏保守的假设。目前大多数美国在线券商（如 Robinhood、Schwab、Fidelity）已经实行零佣金交易，实际成本主要来自买卖价差（bid-ask spread）。QQQ 这种高流动性 ETF 的价差通常只有 0.01-0.02%，加上微小的滑点，总成本远低于 0.1%。但对于 PLTR 这种中盘股，价差可能达到 0.03-0.05%，在快速波动时滑点可能更大。如果策略应用于小盘股或新兴市场，0.1% 可能偏乐观，需要提高到 0.3-0.5%。总的来说，0.1% 对这三个标的是合适的。

**41. 为什么要和买入持有做对比？**

买入持有是最低成本、最低复杂度的策略——你只需要买一次，然后什么都不做。如果一个复杂的 ML 策略跑不赢买入持有，那就说明你付出的所有努力（数据获取、特征工程、模型训练、交易执行）没有创造任何价值（alpha），你还不如把时间花在别的事情上。买入持有也是金融理论中"有效市场假说"的实践体现——如果市场是完全有效的，任何策略都不可能持续战胜买入持有。你的 ML 策略能跑赢买入持有的程度，就是你的模型捕捉到的市场非有效性（inefficiency）的大小。

**42. 回测表现好 = 未来表现好吗？**

绝对不等于。回测表现好只能说明模型在历史数据上发现了一些模式，但不保证这些模式在未来会重复。至少有三个风险：(1) 过拟合——模型可能记住了噪音而不是信号，换一段数据就失效。(2) 制度变化——市场结构在不断演变（如量化交易占比增加、监管政策变化），旧的规律可能消失。(3) 信号衰减——如果某个信号被广泛发现和利用，它会逐渐被交易行为消化掉（如 2000 年前的动量效应远强于现在）。合理的做法是：把回测当作"必要但不充分"的筛选条件——回测差的策略肯定不行，回测好的策略也不一定行，需要在 paper trading（模拟盘）中进一步验证。

### 实际交易相关（43-50）

**43. 模型预测的概率和真实概率一样吗？**

不一样。模型输出的 `predict_proba` 是一个"得分"，反映模型的置信度，但不是真正的概率。举例：模型说"上涨概率 60%"，但在历史上模型给出 60% 的所有预测中，实际上涨的比例可能是 55% 或 65%。这就是概率校准（probability calibration）的问题。树模型的概率通常没有经过校准——因为树模型的预测基于训练样本中正例的比例，在叶子节点样本量小的时候会不准确。如果需要精确的概率估计（比如用于凯利公式计算最优仓位），可以用 Platt Scaling 或 Isotonic Regression 做校准。对于这个项目的简单分档策略（>0.6 满仓、>0.5 半仓），概率校准的影响较小。

**44. 小资金（比如 1000 美元）适合怎么用这个模型？**

1000 美元适合买 ETF（如 QQQ），因为 ETF 可以买小数股（fractional shares），而且流动性好、价差小。具体操作：(1) 每两周看一次模型预测；(2) 如果 ensemble 概率 > 0.6，全部投入 QQQ；(3) 0.5-0.6 之间投入一半，剩余放货币基金；(4) < 0.5 全部持有现金。小资金要特别注意：不要频繁交易，因为每次交易的固定成本（哪怕是零佣金，还有税务处理成本）对小账户的影响更大。也不要加杠杆——模型的信号很弱，杠杆会放大损失。把这当作学习工具，而不是赚钱工具。

**45. 为什么不建议 all-in？**

All-in 意味着把全部资金投入单一标的，这违反了投资学最基本的分散化原则。原因有三：(1) 模型可能错——即使概率显示 60%，仍有 40% 的可能是错的，all-in 在错误时损失 100% 的本金。(2) 黑天鹅风险——模型无法预测的极端事件可能导致单一标的短时间暴跌 30-50%（如 PLTR 在 2022 年跌了 65%）。(3) 心理压力——全仓持有一只股票时，每一次下跌都让你坐立不安，很容易在最差的时机恐慌性卖出。专业的风险管理建议单一标的最多占总仓位的 20-30%，并且根据 ATR（波动率）动态调整——波动大的股票仓位小，波动小的 ETF 仓位大。

**46. QQQ/GOOGL/PLTR 三个标的的风险特征差异？**

这三个标的代表了不同的风险谱系。QQQ 是纳斯达克 100 ETF，持有约 100 只科技股，天然分散化，波动率中等（年化约 20-25%），是最"安全"的选择。GOOGL 是 Alphabet 单一股票，市值超万亿美元，属于"大盘科技股"，波动率略高于 QQQ（约 25-30%），但受到单一公司事件（财报、反垄断）的影响更大。PLTR 是帕兰提尔科技，市值较小（约 500-1000 亿美元），增长速度快但估值高，波动率最大（年化 40-60%），单日涨跌 5-10% 是家常便饭。从风险管理角度，QQQ 适合保守型投资者，PLTR 适合高风险偏好的投资者，GOOGL 居中。

**47. 模型多久需要重新训练？**

建议每 1-3 个月重新训练一次。理由如下：(1) 新数据积累——每个月新增约 22 个交易日的数据，3 个月就是 66 行新样本，能让模型学到最新的市场模式。(2) 特征分布漂移（concept drift）——市场环境在变化，比如 2022 年是加息周期，2024 年可能是降息周期，特征和目标之间的关系会改变。(3) 宏观数据更新——VIX、TLT 等宏观因子在不同时期的重要性不同。重新训练时应该使用最新的数据，更新 TRAIN_END 和 VAL_END，并检查特征重要性是否发生了显著变化。如果发现模型的实盘表现明显下降（如连续一个月预测准确率低于 48%），应立即重新训练。

**48. 市场环境变化（如加息周期）会影响模型吗？**

会，而且影响很大。这叫做"制度转换"（regime change）。在低利率环境下（2020-2021），市场偏好成长股，VIX 通常较低，TLT 相对稳定。在加息周期（2022-2023），高估值的科技股被大幅杀估值，VIX 飙升，TLT 暴跌。这意味着特征和目标之间的关系发生了根本性变化——同样的"RSI=30"，在牛市中是抄底信号，在熊市中可能是"下跌中继"。项目通过加入宏观因子（VIX/TLT/UUP）部分缓解了这个问题——模型可以"感知"到当前的宏观环境。但最彻底的解决方案是定期重新训练，让模型在最新的市场环境中学习。

**49. 这个模型能用于日内交易吗？**

不能，原因有三。第一，模型的预测周期是 10 个交易日，所有技术指标都基于日线数据计算——RSI(14) 代表 14 天的动量，不是 14 分钟的。第二，日内交易需要 tick 级或分钟级数据，yfinance 不提供高频实时数据（有 15 分钟延迟）。第三，日内交易的信号来源完全不同——需要关注订单簿（order book）、tick 级成交量、微观结构等，这些都不在本项目的特征集中。如果想做日内交易，需要完全不同的技术栈：实时数据 API（如 Alpaca）、低延迟执行系统、以及更适合高频数据的模型（如在线学习、强化学习）。

**50. 如果想继续深入量化交易，下一步学什么？**

建议分三个方向深入：

**基础理论**：阅读 Marcos Lopez de Prado 的《Advances in Financial Machine Learning》，这是金融 ML 的圣经，详细讲解了前视偏差、特征重要性、元标签等核心概念。同时学习 Ernest Chan 的《Algorithmic Trading》了解实际交易的工程问题。

**技术能力**：学习 Backtrader 或 Zipline 等专业回测框架（比这个项目的回测引擎强大得多）。学习 Alpaca API 做 paper trading（模拟盘）。掌握数据库（PostgreSQL + TimescaleDB）存储大量历史数据。了解 Docker 部署和定时任务调度。

**高级话题**：学习另类数据（alternative data）——新闻情感分析、卫星图像、社交媒体信号。了解组合优化（portfolio optimization）和风险管理框架。研究强化学习在交易中的应用。考虑学习 C++ 或 Rust，因为真正的高频交易对延迟极为敏感。

## 第五部分：项目的局限性与改进方向

### 数据量偏少的影响

项目每个标的只有约 1300 个样本，这带来了几个问题：
- **交叉验证的方差大**：TimeSeriesSplit 的每个 fold 只有两三百个样本作为验证集，评估指标的波动性较大。
- **低频模式学不到**：极端市场事件（如 VIX > 40 的恐慌期）在数据中可能只出现 1-2 次，模型无法从中学到可靠的规律。
- **过拟合风险**：30 个特征 / 1300 个样本 = 特征/样本比约 1:43，虽然在可接受范围内，但不算宽裕。

**改进方向**：
- 将 `DATA_START` 提前到 2015 年或更早（对 QQQ 和 GOOGL 来说有足够历史数据），增加到 2000-2500 个样本。
- 通过多个标的共同训练一个通用模型（multi-asset model），共享学到的模式。
- 使用时间序列数据增强技术（如窗口滑动、合成少数类样本 SMOTE 的时间序列变体）。

### 没有基本面数据和新闻情感的局限

项目完全基于技术面（价格和成交量衍生的指标）和宏观面（VIX/TLT/UUP），缺少两类重要信息：

**基本面数据**：市盈率（P/E）、营收增速、利润率等。特别是在财报季（每季度），股价的大幅波动主要由基本面驱动，纯技术面模型在这几天的预测能力会显著下降。

**新闻/情感数据**：美联储声明、CEO 离职、产品发布等新闻事件是短期股价的主要驱动力。可以通过 NLP 模型分析新闻标题的情感倾向（看涨/看跌），作为额外的特征。

**改进方向**：
- 接入 SEC 财报数据（通过 `sec-edgar` 库）或 Financial Modeling Prep API。
- 使用 FinBERT 或 GPT 对金融新闻做情感分析，生成日度情感得分作为特征。
- 加入分析师评级变化（upgrade/downgrade）作为信号。

### 模型无法应对突发事件

机器学习模型本质上是"历史规律的外推器"，它假设未来和过去具有相似的统计结构。以下情况会导致模型失效：
- **黑天鹅事件**：疫情、战争、金融危机等前所未有的事件。
- **政策突变**：如美联储突然加息 100 个基点，或某国宣布禁止做空。
- **市场微结构变化**：如 GameStop 事件中散户通过社交媒体协同操作，打破了传统的技术分析逻辑。

**改进方向**：
- 加入风控模块：无论模型怎么说，单日亏损超过 3% 就强制止损。
- 实时监控 VIX 水平：VIX > 35 时自动降低仓位或停止交易。
- 设计"模型信心检测器"：当最新数据的特征分布偏离训练数据太远时（用马氏距离度量），自动发出警告。

### 其他改进方向

1. **概率校准**：用 `CalibratedClassifierCV` 校准模型输出的概率，使其更接近真实概率。
2. **特征选择**：基于特征重要性做递归消除（RFE），移除噪音特征。
3. **动态权重**：集成模型的权重不必固定——可以根据最近 N 天的预测表现动态调整（表现好的模型权重上升）。
4. **多时间尺度**：同时训练 5 天、10 天、20 天前瞻的模型，综合多个时间尺度的信号。
5. **在线学习**：不必每次全量重新训练，而是用增量学习（online learning）在新数据到来时更新模型。
6. **仓位优化**：用凯利公式（Kelly Criterion）根据模型概率和历史赔率计算数学最优仓位，替代当前的简单分级策略。
7. **Paper Trading**：接入 Alpaca 或 Interactive Brokers 的模拟盘 API，用真实的市场数据验证策略，但不用真钱。
